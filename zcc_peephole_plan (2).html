static long long t5_eval_const(Node *n, int *ok) {

The Self-Referential Loop
ZCC's codegen_expr() is itself a deeply recursive function — it calls itself on every sub-expression. With T6 active, when ZCC compiles a tail-recursive C function, ZCC's own recursive descent is simultaneously being optimized by the very TCO machinery it implements. The compiler rewrites the recursion it uses to do the rewriting. The Golden Rule holds because TCO only fires when node->lhs->func_name == cc->current_func — ZCC's own calls to codegen_expr() are mutual recursion (not strict tail recursion to self), so they pass through the standard call path untouched.

Assembly Sequence — Standard Variant (addCorr == 0)
movabsq $M, %rcx
Load 64-bit magic constant. Must use movabsq (64-bit immediate) — movq only sign-extends 32-bit immediates.
imulq %rcx
128-bit signed multiply: rdx:rax = x × M. The high 64 bits (rdx) contain our approximation of x / d.
[sarq $s, %rdx]
Optional post-shift. Only emitted if s > 0. Shifts the approximation right to align to the exact quotient range.
movq %rdx, %rax
Move high half (our result) to %rax for sign correction step.
sarq $63, %rax
Propagate sign bit: 0 if result ≥ 0, −1 if result < 0. Branchless — no conditional jump.
subq %rax, %rdx
Subtract sign: adds 1 for negative results (since −1 subtracted = +1 added). Corrects truncation-toward-zero per C99.
movq %rdx, %rax
Final result in %rax. 7 instructions replacing cqo + idivq. ~8–10 cycles vs 35–90 cycles.
Assembly Sequence — Overflow Variant (addCorr == 1)
Fires when M ≥ 2⁶³ (magic constant doesn't fit in signed 64-bit). The imulq product is off by x × 2⁶⁴, requiring a correction. Affects d = 100, 255, and many others. Try them in the magic number lab widget.
movq %rax, %r10
Save dividend x before the multiply clobbers %rax.
movabsq $M, %rcx
Load wrapped magic constant (negative as signed 64-bit).
imulq %rcx
128-bit multiply. Product is x × (M − 2⁶⁴) in rdx:rax. Undershoots by x × 2⁶⁴.
addq %r10, %rdx
Correction: add x to rdx. This is equivalent to adding x × 2⁶⁴ to the 128-bit product, restoring the correct high half.
[sarq $s, %rdx]
movq/sarq $63/subq/movq
Same sign-correction tail as standard variant.

Bug #1 — AST Evaluation Order Collapse RESOLVED
Tier: T1 ND_ASSIGN  |  PRIME H₀: 0.95  |  Risk: CATASTROPHIC
During ND_ASSIGN refactor, RHS was evaluated before LHS. In buf[i++] = read_char(cc), the read_char side-effect advanced the file position before buf[i++] was evaluated, breaking the parser's token stream when compiling itself.
Fix: Restored default C sequential eval order — LHS address computed and pushed first, then RHS evaluated.

Bug #2 — ILP32 Lexer Wraparound Matrix RESOLVED
Tier: PREREQ  |  PRIME H₀: 0.85  |  Risk: CRITICAL
ZCC uses ILP32 (long = 32-bit). GCC uses LP64 (long = 64-bit). Hex constant 0xFFFFFFFF lexed to 4294967295 under GCC and -1 under ZCC — two compiler stages diverging textually on the same constant. Also affected: bounds checks against -2147483648LL.
Fix: Surgical (int)val cast in part2.c lexer tk_val assignment — force 32-bit truncation in GCC stage too.
part2.c Attractor collision: two compilers, same seed, divergent topology
Bug #3 — T2 B6 Pointer Scaling Inversion (Segfault) RESOLVED
Tier: T2 B6  |  PRIME H₀: 0.90  |  Risk: CATASTROPHIC
B6 register swap moves LHS to %rcx, RHS to %rax. For pointer addition, the index (RHS, in %rax) must be scaled by element size — not the pointer address (LHS, in %rcx). Initial code scaled %rcx (the address), causing array traversals to write wildly OOB. GDB trace: crash in init_keywords on kw_word[i] = "int".
Fix: Swapped imulq target registers in part4.c — scale %rax (index), never %rcx (pointer address).
part4.c Small perturbation → global memory corruption
Bug #4 — 64-Bit Lexer Singularity RESOLVED — T4-D
Tier: T4-D integration  |  PRIME H₀: 0.88  |  Risk: CRITICAL
The Bug #2 fix ((int)val cast) became the enemy of T4-D. Magic constants like 6148914691236517206LL were being silently truncated to 1431655766 — top 32 bits amputated — causing hundreds of assembly diffs. Secondary root cause: the lexer's string accumulator used long val, which is 32-bit under ILP32, crushing multi-digit LL constants during token parsing.
Fix (two-part): (1) Promote accumulator long → long long. (2) Add is_ll suffix lookahead — bypass 32-bit cast for LL-suffixed literals, preserve it for normal integers.
